## Phase H.3 — Healing the Geography & Image Layers (Enterprise, Surgical, Tenant-Aware)

### Why this plan exists (verified facts, not assumptions)

I queried the live database and inspected every relevant file before writing this. The findings explain every single symptom you reported:

| Symptom you reported | Root cause confirmed in the code/DB |
|---|---|
| "No mini-maps visible" | `srangam_article_pins` has **0 rows** and `srangam_gazetteer` has 32 rows. The map card only renders when `article.pins.length > 0`. We shipped the scaffolding in Phase H.2 but the **backfill edge function was never invoked**, so every article still has zero pins. |
| "Where do I backfill from?" | There is no admin UI button for `backfill-article-pins`. Only `generate-article-og` and `backfill-bibliography` are wired in `DataHealth.tsx`. |
| "OG generation should use Gemini, not OpenAI" | `supabase/functions/generate-article-og/index.ts` still calls **`dall-e-3`** with `OPENAI_API_KEY`. The tenant-aware `_shared/ai-provider.ts` we built in H.2 is only used by the pin backfill, not by image generation. 32 articles already have OG images stored in GDrive — all generated by DALL-E. |
| "Retire old images, regenerate via admin" | No `og_image_status` column, no archive table, no "regenerate one" UI. Current admin flow only fills *missing* images; it never replaces. |
| "Maps & Data page is a joke" | `src/pages/MapsData.tsx` reads `src/data/cosmic_ocean/ports.json` (a static 8-port file). It has zero connection to `srangam_gazetteer`, `srangam_article_pins`, or the 42 published articles. The "Loading map data…" you screenshotted is `OceanMap` waiting on a static fetch, not on article geography. |
| "Evidence not driving pins" | 79 evidence rows exist but **0 have lat/lon**. Phase H.2's confidence-A path can never fire on real data without a geocoding step. |

### Enterprise software we are explicitly modelling on

These are real, mature products in the same problem-space. Each row maps a feature we will copy (not a vague aspiration).

| Product | Pattern we adopt | Where we apply it |
|---|---|---|
| **Pelagios / Recogito** (classics gazetteer + annotation) | Gazetteer + per-document place pins with confidence tiers; all linkable by URI | `srangam_gazetteer` + `srangam_article_pins` (already exist, need population) |
| **GeoNames / WHGazetteer** | Canonical place + alternate-name index with `name_variants[]` and external IDs | Already in our schema (`name_variants`, `external_refs`); we'll add a `geonames_id` lookup |
| **ArchesProject (Getty/WMF)** | Resource-Map page that aggregates every geo-tagged record on one Leaflet/MapLibre canvas with facet filters (period, theme, confidence) | New `MapsData.tsx` "Article Geography" panel |
| **Cloudinary / Imgix** | Asset records with status (`active`, `superseded`, `deleted`), version numbers, idempotent regenerate API | New `srangam_media_assets` table + admin "Regenerate / Retire" actions |
| **Vercel OG / Next OG image** | Provider-pluggable image generation behind one edge function with cost telemetry | Make `generate-article-og` use the existing `_shared/ai-provider.ts` (Gemini → OpenAI fallback) |
| **Algolia Places / Mapbox Geocoding** | Server-side text-to-coord resolution with tier-A confidence when matched in a gazetteer | New `geocode-evidence` step in the backfill function |

DOES NOT touch: the Phase H.2 schema, the article resolver, the lazy `ArticleMiniMap` component, the gdrive-image-proxy, the cards layout, routing, theming, narration, or any working article pipeline. All edits are additive or behind feature-flag-equivalent guards (empty data ⇒ no UI change).

DOES change exactly the listed files. Every change is reversible by reverting the file.

## Step-by-step changes

### Step 1 — Run the backfill (data, not code) so existing articles light up

Operator action only — no schema change. From the new admin button (Step 4) we invoke `backfill-article-pins` with `{all_published: true, limit: 50}`. Idempotent — safe to re-run.

Expected outcome on first run: 25-40 of 42 published articles get 1-5 pins each from regex + AI NER against the gazetteer's 32 places. Mini-map cards then appear automatically because `article.pins.length > 0` is already wired.

### Step 2 — Make OG generation tenant-aware (Gemini-first, retire DALL-E path)

Edit `supabase/functions/generate-article-og/index.ts`:

1. Replace the OpenAI DALL-E 3 call with `callImage()` from `_shared/ai-provider.ts` (extend the helper — see Step 3).
2. Keep the existing GDrive upload + RS256 JWT code untouched (it works).
3. Keep the `themeColors` palette and prompt template untouched (visual consistency).
4. Filename convention upgraded to `og-{slug}-v{version}-{yyyymmdd}.png` so retired versions are auditable in GDrive.
5. Return `{provider, model, cost_usd, gdrive_file_id, version}` for telemetry.

### Step 3 — Extend `_shared/ai-provider.ts` with `callImage()`

Add one function (≈40 lines), no breaking changes:

- Primary: `gemini-3-pro-image-preview` (Nano Banana Pro) using `GEMINI_API_KEY` against `https://generativelanguage.googleapis.com/v1beta/models/.../generateContent` — context-aware, supports the title + theme prompt natively.
- Fallback: `gpt-image-1` if Gemini returns 4xx/5xx (DALL-E 3 sunset path; OpenAI key already present).
- Returns `{bytes: Uint8Array, provider, model, cost_usd, latency_ms}` matching the existing text-NER shape.
- Gateway bypassed — uses the customer's own keys, exactly as you instructed.

### Step 4 — New admin "Geography & Media" tab (the missing UI)

Backed by three edge-function calls already-existing or added in this plan:

Surgical: this page is additive, not a replacement. Existing `DataHealth.tsx` keeps its bulk OG button (label changed to "(legacy bulk)" — operator can deprecate at leisure).

### Step 5 — Image lifecycle table (so retire/replace is auditable)

One migration adding `srangam_media_assets`:

```text
id uuid pk
article_id uuid fk srangam_articles
kind text  -- 'og_image' | 'hero' | future use
provider text          -- 'gemini' | 'openai'
model text             -- 'gemini-3-pro-image-preview', etc.
gdrive_file_id text
gdrive_share_url text
version int            -- 1, 2, 3 ...
status text            -- 'active' | 'superseded' | 'retired'
prompt_hash text       -- so re-runs with identical prompt are idempotent
cost_usd numeric
created_at timestamptz default now()
retired_at timestamptz
```

Plus: add `og_image_version int default 1` and `og_image_status text default 'active'` to `srangam_articles` for fast read-side joins. RLS: admin-only writes, public read of `active` rows.

`generate-article-og` writes a new row each time; `retire-og-image` flips the previous row to `superseded` and updates `srangam_articles.og_image_url` + `og_image_version` atomically. Old GDrive files are **not** deleted automatically — they're marked `retired_at` so an operator can sweep them on a schedule. (This matches Cloudinary's "soft retire" pattern; deletion is irreversible and should be deliberate.)

### Step 6 — Heal the Maps & Data page (the real fix you asked for)

Edit `src/pages/MapsData.tsx`:

1. **Keep** the existing `OceanMap`, monsoon, ports cards (no regression — the page still works for visitors who land on `/maps-data?layer=ports`).
2. **Add** a new top-of-page panel: "Article Geography — Indian Ocean Atlas" — a single Leaflet/MapLibre map driven by `srangam_article_pins ⨝ srangam_gazetteer ⨝ srangam_articles (status='published')`.
3. Each marker = one place; popup lists every article that pins it, with tag chips and a click-through to `/articles/{slug}`.
4. Facet sidebar: theme, era_tag, confidence (A/B/C). All client-side filtering on a payload of <2,000 rows — no extra round-trips.
5. Reuses `ArticleMiniMap`'s lazy-Leaflet pattern; adds ≈0kB to the rest of the site.
6. New hook `useArticleGeography()` — single Supabase query (one round-trip, indexed) cached by React Query for 5 min.

This is what you described: "the geographical spread and references of the various articles and their keywords and coordinates." It's a one-page Arches-style resource map, not a dashboard.

### Step 7 — Geocode evidence to unlock confidence-A backfill

Edit `supabase/functions/backfill-article-pins/index.ts` to add a pre-pass: for any `srangam_article_evidence.place IS NOT NULL AND latitude IS NULL`, query `srangam_gazetteer` by `canonical_name` ILIKE or `name_variants &&`. On hit, write `latitude/longitude` back to the evidence row (one cheap UPDATE) and emit a confidence-A pin. No external geocoding API needed for the 79 existing rows — the gazetteer already covers the 32 places they reference.

## Risk & rollback

- Every code path has an empty-data fallback already (proven: today the maps card simply doesn't render). If Step 1 errors out, nothing visible breaks.
- Image retire is soft (status flip), not destructive. Wrong retirement is reversed by an UPDATE.
- Provider swap is behind try/Gemini → catch/OpenAI, so a Gemini outage doesn't stop image generation.
- Migration is additive — no column drops, no data rewrites.

## Performance budget (measured against current bundle)

- New admin page: lazy-loaded route, +0kB on public site.
- `ArticleAtlasMap`: dynamic Leaflet import (same pattern as `ArticleMiniMap`), +0kB until user visits `/maps-data`.
- `useArticleGeography`: one Supabase round-trip (~1-3kB JSON), 5-min cache.
- OG generation: same per-article cost (Gemini Pro Image ≈ $0.04, OpenAI fallback ≈ $0.04). No change on the visitor critical path.
